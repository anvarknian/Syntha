# Contributing to Syntha

Thank you for wanting to contribute! This document explains how to contribute code, documentation, issues, and design feedback to the Syntha project.

## Table of contents
- How to report issues
- How to propose changes (PRs)
- Code style and tests
- Branching and commit guidelines
- Review process
- Community conduct

## Reporting issues
- Search existing issues before opening a new one.
- Provide a clear title and a reproducible description.
- Include environment details and steps to reproduce.

## Proposing changes (Pull Requests)
- Fork the repository and create a feature branch from `main`.
- Keep PRs small and focused — one logical change per PR.
- Describe the problem, design, and migration notes in the PR description.
- Link related issues and include screenshots or logs when helpful.

## Code style and tests
- Follow the repository's existing style and conventions.
- Run tests locally before opening a PR.
- Add unit tests for bug fixes and new features when applicable.

## Branching and commit guidelines
- Branch name format: `type/short-descriptor` (e.g. `feat/browser-cluster` or `fix/replay-bug`).
- Use atomic commits with clear messages. Use present-tense short summary.
- Squash or rebase before merging when requested by reviewers.

## Review process
- PRs should include a clear summary and testing steps.
- Maintainership will review for design, tests, security, and style.
- Be responsive to reviewer feedback; iterate on changes as requested.

## Documentation
- Keep docs next to implementation when possible (e.g., `/docs`).
- Update `README.md` and relevant docs for publicly visible changes.

## Community Code of Conduct
- Be respectful, constructive, and collaborative.
- No harassment, hateful language, or personal attacks.
- Report policy violations to project maintainers privately.

Thank you — contributions make Syntha better.
